# Archive Persistence Design

**Status:** Design proposal
**Issue:** sapling-2da7
**Date:** 2026-03-04

## Problem

When a swarm agent is restarted (e.g., by overstory after a crash, timeout, or manual nudge), all context pipeline state is lost. `SaplingPipelineV1` holds its state entirely in memory:

- `operations: Operation[]` — the full operation registry (active, completed, compacted, archived)
- `activeOperationId: number | null` — which operation is currently in progress
- `nextOperationId: number` — monotonic counter for operation IDs

On restart, the agent begins with an empty pipeline. It has no knowledge of what files it modified, what operations succeeded or failed, or what working memory summaries were generated. This means:

1. **Lost working memory** — archived operation summaries (injected into the system prompt) disappear
2. **Duplicate work** — the agent may re-explore files it already read or re-attempt changes it already made
3. **Lost error context** — unresolved errors from previous operations are forgotten
4. **Broken dependency chains** — dependsOn relationships between operations are lost

## What to Persist

### Core state (required)

The minimal set needed to restore a useful working memory:

```typescript
interface PersistedPipelineState {
  /** Schema version for forward compatibility. */
  version: 1;
  /** ISO timestamp of last write. */
  lastWritten: string;
  /** Agent identity (for conflict detection in multi-agent scenarios). */
  agentId: string;
  /** Next operation ID counter (to avoid ID collisions on restart). */
  nextOperationId: number;
  /** Persisted operation summaries. */
  operations: PersistedOperation[];
}

interface PersistedOperation {
  id: number;
  status: "completed" | "compacted" | "archived";
  type: OperationType;
  outcome: "success" | "failure" | "partial";
  /** File paths touched by this operation. */
  files: string[];
  /** File paths created/modified (artifacts). */
  artifacts: string[];
  /** IDs of operations this one depends on. */
  dependsOn: number[];
  /** Score at time of persistence (informational only — re-evaluated on reload). */
  score: number;
  /** Compact summary text (from templates.ts). */
  summary: string | null;
  /** Turn range (for ordering). */
  startTurn: number;
  endTurn: number;
}
```

### What NOT to persist

- **Raw turn messages** — large, quickly stale, and already summarized by the compact stage. Summaries capture essential info.
- **Active operations** — incomplete at crash time. Discard and let agent re-discover state on restart.
- **Boundary signals/weights** — ephemeral per-turn computation.
- **Budget utilization** — recomputed each pipeline cycle from operations list.

### Design rationale

Archived and compacted operations already have summaries generated by `renderCompactSummary()` in `templates.ts`. These summaries are what gets injected into the system prompt as working memory. By persisting operation metadata + summaries, a restarted agent can immediately reconstruct its working memory section without needing raw message history.

## File Format

### Location

```
.sapling/pipeline-state.<agentId>.json
```

Per-agent filenames avoid write contention in swarm scenarios (multiple agents sharing a project). Each agent owns its file exclusively.

Rationale:
- `.sapling/` is the established project-level config directory (used by `session.jsonl`, `config.yaml`, `guards.json`)
- Single JSON file (not JSONL) because the state is a snapshot, not an append-only log
- Named `pipeline-state` rather than `archive` because it contains more than just archive entries

## When to Write

### Strategy: write on operation finalization + graceful shutdown

1. **On operation boundary** — when `ingestTurn()` detects a boundary and calls `finalizeOperation()`, persist after the full pipeline cycle completes. Captures each completed operation incrementally.

2. **On graceful shutdown** — when the loop exits normally (task complete or max turns), persist final state. Captures any in-progress operation that was never finalized (persisted as `completed` with inferred outcome).

3. **NOT every turn** — writing every turn would create excessive I/O for minimal benefit. Operations typically span 3-8 turns, so writing on finalization is frequent enough.

### Implementation sketch

```typescript
// In SaplingPipelineV1
private dirty = false;

process(input: PipelineInput): PipelineOutput {
  const prevOpCount = this.operations.length;
  // ... existing pipeline logic ...

  // Mark dirty if new operations were finalized
  if (this.operations.length > prevOpCount ||
      this.operations.some(op => op.status !== prevStatuses.get(op.id))) {
    this.dirty = true;
  }

  if (this.dirty) {
    this.persistState();
    this.dirty = false;
  }

  return output;
}

/** Called by loop.ts on graceful shutdown. */
shutdown(): void {
  const activeOp = this.operations.find(op => op.id === this.activeOperationId);
  if (activeOp) {
    activeOp.status = "completed";
    activeOp.outcome = inferOutcome(activeOp);
    if (activeOp.summary === null) {
      activeOp.summary = renderCompactSummary(activeOp);
    }
  }
  this.persistState();
}
```

### Write mechanism

Use `writeFileSync` with temp-file-then-rename for atomicity:

```typescript
private persistState(): void {
  const state: PersistedPipelineState = { /* serialize ops */ };
  const filePath = join(this.projectDir, ".sapling",
    `pipeline-state.${this.agentId}.json`);
  const tmpPath = filePath + ".tmp";
  writeFileSync(tmpPath, JSON.stringify(state, null, 2) + "\n");
  renameSync(tmpPath, filePath);
}
```

## When to Read (Restore)

On pipeline construction, check for the agent state file:

1. Parse and validate the `version` field
2. Restore `nextOperationId` (critical to avoid ID collisions)
3. Convert persisted operations to `Operation` objects (empty `turns` arrays)
4. All restored operations get status `archived` — without raw turn messages, they can only contribute as working memory entries

```typescript
private loadPersistedState(projectDir: string): void {
  const filePath = join(projectDir, ".sapling",
    `pipeline-state.${this.agentId}.json`);
  if (!existsSync(filePath)) return;

  try {
    const raw = readFileSync(filePath, "utf-8");
    const state: PersistedPipelineState = JSON.parse(raw);
    if (state.version !== 1) return;

    this.nextOperationId = Math.max(this.nextOperationId, state.nextOperationId);
    for (const persisted of state.operations) {
      this.operations.push({
        ...persisted,
        status: "archived",
        files: new Set(persisted.files),
        tools: new Set(),
        turns: [],
      });
    }
  } catch {
    // Corrupted file — log warning, proceed without state
  }
}
```

## Conflict Handling for Concurrent Agents

### Per-agent files (primary strategy)

Each agent writes to `.sapling/pipeline-state.<agentId>.json`. No write contention, no locking needed. A restarted agent reads only its own state.

### Cleanup

Combine two strategies:
- **TTL-based:** On read, ignore files with `lastWritten` older than 24 hours
- **Delete on graceful exit:** Agent removes its state file on clean shutdown (persist only for crash recovery)

This handles crashes (stale file expires after TTL) and normal operation (file removed immediately).

## Migration and Compatibility

- **version field** allows schema evolution. Unknown versions are skipped (agent starts fresh).
- **gitignore:** Add `.sapling/pipeline-state*.json` — runtime data, not to be committed.
- **No .gitattributes merge strategy needed** since files are gitignored.

## Summary of Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| What to persist | Op metadata + summaries (no raw messages) | Summaries capture essential info; raw messages are large and stale |
| File format | JSON snapshot | State is a point-in-time snapshot, not append-only |
| File location | `.sapling/pipeline-state.<agentId>.json` | Per-agent avoids contention in swarm |
| When to write | On op finalization + graceful shutdown | Balances I/O cost vs. data freshness |
| When to read | Pipeline construction | Earliest restore point |
| Active operations | Not persisted | Incomplete state is hard to reason about |
| Conflict handling | Per-agent filenames | Simple, no locking needed |
| Cleanup | TTL (24h) + delete on graceful exit | Handles crashes and normal exits |

## Open Questions

1. **Cross-agent working memory:** Should a restarted agent read other agents archived operations for broader context? Deferred to follow-up.

2. **Incremental persistence:** JSONL event log vs. JSON snapshot. Snapshot is simpler and sufficient for typical session lengths (< 200 turns).

3. **State file size limits:** For very long sessions (100+ ops), cap at 50 most recent operations. Unlikely issue in practice (~200 bytes per archived op summary).
